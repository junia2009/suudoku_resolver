# ナンプレ絶対解くマン — Sudoku Solver (ver 1.9.2)

カメラ撮影・ギャラリー読み込みから数独を自動解読するウェブアプリ（PWA）です。  
GitHub Pages で動作し、インストール不要でスマートフォン・PC から利用できます。

## 🔗 公開URL
`https://<あなたのGitHubユーザー名>.github.io/<リポジトリ名>/`

---

## 機能

| ステップ | 処理内容 |
|----------|----------|
| 1. 画像入力 | カメラ撮影 または ギャラリーから画像選択 |
| 2. 歪み補正 | OpenCV.js で数独グリッドを自動検出・射影変換。検出失敗時は4隅を手動タップで指定可能 |
| 3. 数字認識 | Tesseract.js OCR (トリプルPSM + 複数前処理 + Otsu二値化 + 連結成分切り出し + 構造解析) で 1〜9 を認識。誤認識はタップして修正可能 |
| 4. 解答表示 | 制約伝播 + バックトラッキング (MRV) で解を求めて表示 |
| 5. 解法解説 | 解答画面の「解説を見る」ボタンで、どの順番・どの手法で数字が埋まるかをステップごとに解説。自動再生にも対応 |

## 技術スタック

- **フロントエンド**: HTML / CSS / Vanilla JavaScript (モジュール構成) / PWA (Service Worker)
- **画像処理**: [OpenCV.js 4.12.0](https://cdn.jsdelivr.net/npm/@techstark/opencv-js@4.12.0-release.1/dist/opencv.js) — CLAHE・ガウシアンブラー・適応的閾値・形態学的処理・輪郭検出・射影変換・Hough 線検出
- **数字認識**: [Tesseract.js 5](https://tesseract.projectnaptha.com/) — トリプル PSM (7/10/13) OCR エンジン + 構造解析 (穴カウント)
- **ソルバー**: 制約伝播 (Naked Singles + Hidden Singles) + バックトラッキング (MRV ヒューリスティック) + 解法ステップ記録
- **ホスティング**: GitHub Pages (静的サイト)

## 数字認識ロジックの詳細

数字認識は `js/digitRecognizer.js` (v1.8.0+) が担当します。  
Tesseract.js を使ったトリプル PSM アプローチで、各セルを多数決方式で判定します。

### 1. OCR ワーカーの初期化

起動時に 3 つの Tesseract.js ワーカーを並列で初期化します。

| ワーカー | PSM | 概要 |
|----------|-----|------|
| `_worker7`  | PSM 7 — Single text line | 単一テキスト行として認識 |
| `_worker10` | PSM 10 — Single character | 1 文字だけを含む画像として認識 |
| `_worker13` | PSM 13 — Raw line | ページセグメンテーションなしで raw テキストとして認識 |

すべて `tessedit_char_whitelist: '123456789'` を設定し、数字以外の誤検出を防ぎます。

### 2. 空セル判定 (`_isEmpty`)

OCR 前に各セル画像が「空白（数字なし）」かどうかを判定し、空白セルは OCR をスキップします。

1. セル中央部（上下左右 20% マージンを除いた領域）のピクセル輝度を集計します。
2. **標準偏差 < 10** → 輝度が均一 → 空白と判定
3. **暗いピクセルの割合 < 0.8%**（閾値 = 平均輝度 × 0.6 未満のピクセル）→ 数字なし → 空白と判定

上記どちらかに該当するセルは `0`（空白）として扱い、OCR をスキップします。

### 3. 前処理バリエーションの生成

空白でないセルには以下の前処理を施した複数のバリエーション画像を用意し、それぞれ OCR を実行します。

| バリエーション | 処理内容 |
|----------------|----------|
| グレー × 3倍拡大 | パディング (30%) 付きで 3 倍に拡大 |
| グレー × 4倍拡大 | パディング付きで 4 倍に拡大 |
| コントラスト強調 × 3倍 | ストレッチング (min-max 正規化) 後に 3 倍拡大 |
| コントラスト強調 × 4倍 | ストレッチング後に 4 倍拡大 |
| CC中心切り出し (グレー) | 連結成分ベースで数字を切り出し・正方形にして 3 倍拡大 |
| CC中心切り出し (コントラスト) | 上記にコントラスト強調を追加 |
| 適応的閾値二値化 × 3倍/4倍 | 輝度反転二値化、拡大 |
| Otsu二値化 × 3倍/4倍 | Otsu法で二値化、拡大 |

**連結成分ベース数字切り出し (`_centerDigitCC`)**: Otsu法で二値化 → 4近傍連結成分ラベリング → ノイズ除去 (面積2%未満) → 最大・中央寄りの成分を選択 → 近接成分をマージ → 正方形にトリミング。  
**従来版フォールバック (`_centerDigit`)**: 連結成分切り出しが失敗した場合に使用。バウンディングボックスで数字を切り出し。

### 4. トリプル PSM での OCR 実行と投票

| ワーカー | 対象バリエーション |
|----------|-------------------|
| PSM 10   | 全バリエーション (最大 10 枚) |
| PSM 7    | グレー 3x/4x、コントラスト 3x、CC中心 3x、Otsu 3x の 5 枚 |
| PSM 13   | グレー 3x/4x、コントラスト 3x、CC中心 3x の 4 枚 |

各 OCR 結果は正規表現 `/^[1-9]$/` に一致した場合のみ有効票とし、各数字について **得票数**・**合計信頼度**・**最高信頼度** を集計します。

### 5. 混同ペア構造解析 (v1.9.2)

OCR 投票で混同しやすいペアの両方に票がある場合、閉領域（穴）の数を数えて構造的に判別します。

| 混同ペア | 構造的特徴 | 判別方法 |
|----------|-----------|----------|
| 3 vs 8   | 8 は 2 穴、3 は 0 穴 | 全領域の穴数をカウント |
| 5 vs 9   | 9 は上部にループ (穴あり)、5 はオープン | 上部 55% の穴数をカウント |

**穴カウント (`_countHoles`)**: Otsu二値化 → 外周背景をフラッドフィルで除去 → 残った未訪問の背景領域 = 穴。面積 0.5% 未満の極小穴はノイズとして無視。

### 6. 最終判定 (多数決 + 信頼度)

得票数でソートし、同票の場合は平均信頼度で比較します。  
以下の条件を満たす場合のみセルに数字を設定します（それ以外は `0`）：

- **得票数 ≥ 2**、または
- **最高信頼度スコア ≥ 60**

### 7. 数独制約による後処理 (`_resolveConflicts`)

OCR 完了後、認識結果が数独のルールに違反していないかチェックします。

- 同じ行・列・3×3 ブロック内に同じ数字が重複している場合、**信頼度が低い方を `0`（空白）に変更**します。
- 矛盾がなくなるまでこれを繰り返します。

## 画像処理ロジックの詳細

画像処理は `js/imageProcessor.js` が担当します。

### グリッド自動検出フロー

1. **グレースケール変換** (`cv.cvtColor`)
2. **コントラスト正規化**: CLAHE (clipLimit=2.0, tileSize=8×8) を優先、失敗時は `equalizeHist` にフォールバック
3. **ガウシアンブラー** (9×9 カーネル)
4. **マルチ閾値戦略**: 以下の 4 パラメータセットで適応的閾値処理を試行し、最初に四辺形が検出できた時点で停止

   | blockSize | C |
   |-----------|---|
   | 11 | 2 |
   | 15 | 3 |
   | 21 | 5 |
   | 7  | 2 |

   各試行で `ADAPTIVE_THRESH_GAUSSIAN_C + THRESH_BINARY_INV` を適用し、3×3 矩形カーネルで `MORPH_CLOSE` を実行。
5. **輪郭検出** (`cv.findContours`): 画像面積の 10% 以上の面積を持つ輪郭のうち、`approxPolyDP` で頂点数が 4 の最大面積の多角形をグリッドとして選択
6. **コーナー整列**: sum (x+y) と diff (x-y) を使って [左上, 右上, 右下, 左下] の順に並べ替え
7. **射影変換** (`cv.getPerspectiveTransform + warpPerspective`): 810×810 px の正方形に変換 (セル 90px)
8. **シャープニング**: ガウシアンブラー (σ=1.2) で平滑化した画像と原画の差分を 1.3 倍で加算
9. **セル分割**: Hough 線検出でグリッド線を精密に検出してセル境界を決定（失敗時は均等 90px 分割にフォールバック）。各セルをグレースケール・適応的閾値二値化・Otsu 二値化の 3 種類で切り出し

自動検出が失敗した場合は、ユーザーが画像上の 4 隅を手動タップで指定できます。

## ソルバーロジックの詳細

ソルバーは `js/sudokuSolver.js` が担当します。

### 制約伝播 (`_propagate`)

バックトラッキング前に以下 2 つのテクニックで確定できるセルを先に埋めます。

- **Naked Singles**: 候補数字が 1 つしかないセルを即確定
- **Hidden Singles**: 行・列・3×3 ブロック内で特定の数字が 1 箇所にしか入れない場合に確定

これを繰り返し適用して探索空間を大幅に縮小します。

### バックトラッキング + MRV ヒューリスティック (`_backtrack`)

空白セルを選ぶ際に **MRV (Minimum Remaining Values)**: 候補数が最小のセルを優先的に選択します。  
候補数が 0 のセルが存在する場合は即座に矛盾と判定しバックトラックします。

### 解法ステップ記録 (`solveWithSteps`)

`solveWithSteps()` は解答に加えて、各セルがどの手法で確定されたかを記録します。  
ステップごとに `{row, col, num, reason, detail}` を配列で返し、解説画面で利用されます。

| reason | 手法 | 説明 |
|--------|------|------|
| `naked_single` | ネイキッドシングル | そのセルに入る候補が1つだけ |
| `hidden_single_row` | ヒドゥンシングル (行) | その行でその数字を置ける場所が1つだけ |
| `hidden_single_col` | ヒドゥンシングル (列) | その列でその数字を置ける場所が1つだけ |
| `hidden_single_box` | ヒドゥンシングル (ブロック) | そのブロックでその数字を置ける場所が1つだけ |
| `backtrack` | 仮定法 (バックトラック) | 候補から1つを仮定して探索 |

## ローカルで動かす

```bash
# Python がある場合
python -m http.server 8080
# または Node.js がある場合
npx serve .
```

ブラウザで `http://localhost:8080` を開いてください。  
**※ `file://` プロトコルでは OpenCV.js / Tesseract.js が動作しません。必ずローカルサーバー経由で開いてください。**

## GitHub Pages へのデプロイ

1. このリポジトリを GitHub にプッシュします。
2. GitHub のリポジトリページ → **Settings** → **Pages**
3. **Source** を `Deploy from a branch` に設定し、ブランチ `main` / `/ (root)` を選択して **Save**。
4. 数分後に公開 URL が表示されます。

## 注意事項

- **HTTPS 必須**: カメラ API (`getUserMedia`) は HTTPS 環境のみで使用可能です。GitHub Pages は自動的に HTTPS で提供されます。
- **Tesseract.js の初回ロード**: 初回実行時に言語データファイルをダウンロードします。ネットワーク環境によっては数秒かかる場合があります。
- **グリッド検出精度**: 明るい場所での撮影、グリッドが画面の大部分を占めるように撮影すると精度が上がります。
- **ブラウザ対応**: Chrome / Safari (iOS 14.5+) / Firefox 最新版を推奨します。
